# GC

## What is it

`GC` (Garbage Collection) là cơ chế JVM tự thu hồi memory của object không còn `reachable` từ `GC roots`.

Mental model ngắn gọn:

- Java không bắt bạn tự `free` memory.
- Nhưng Java cũng không đọc được ý định business của bạn.
- GC chỉ biết một chuyện: object còn đường reference để đi tới hay không.

Vì vậy, GC dọn object đã chết theo `reachability`, không dọn object mà code của bạn vẫn vô tình giữ sống.

## How I used to misunderstand it

Hiểu nhầm phổ biến nhất là “Java có GC rồi thì khỏi lo memory”. Sai ở chỗ GC không sửa được việc thiết kế lifetime kém.

Nếu một `List`, cache, static field, `ThreadLocal`, hoặc listener còn giữ reference, object vẫn còn sống trong mắt JVM dù business đã không cần nữa.

Hiểu nhầm thứ hai là nghĩ `System.gc()` sẽ ép JVM dọn ngay. Thực tế đó chỉ là một hint. JVM có thể bỏ qua, hoặc chạy theo thời điểm riêng của nó.

## How it actually works

JVM bắt đầu từ `GC roots`, rồi đi theo object graph:

- local variables trong stack frame còn sống
- static fields
- active threads
- JNI references và một số runtime roots khác

Object nào còn đi tới được từ các root này thì vẫn `reachable`. Object nào không còn đường đi thì trở thành `eligible for GC`.

Điểm quan trọng là `eligible for GC` không có nghĩa là memory được trả lại ngay ở dòng code kế tiếp. Ở level mental model này, bạn có thể hiểu nó là object đã mất đường `reachable` thông thường từ `GC roots`, nên JVM có thể thu hồi nó khi thấy phù hợp.

### Reachability flow

```text
GC Root
  |
  v
static CACHE -----------> User object
                           ^
                           |
                      local variable

Chỉ cần còn một đường strong reference,
object vẫn còn reachable.
```

### Quick comparison

| Trạng thái | Nghĩa là gì | GC có thể thu hồi ngay không? |
|---|---|---|
| Reachable | Còn đường đi từ `GC root` | Không |
| Eligible for GC | Không còn đường reachable thông thường từ `GC root` | Có thể, nhưng không có thời điểm cố định |
| Collected | Memory đã thực sự được GC reclaim | Đã xảy ra rồi |

### Vì sao GC thường chia thế hệ

Nhiều object trong Java chết rất sớm, ví dụ object tạm trong request, parsing, mapping, hoặc stream pipeline. Vì vậy nhiều JVM/GC collector tối ưu theo giả định “most objects die young”.

Bạn không cần nhớ mọi chi tiết collector để code hằng ngày. Điều nên nhớ là GC luôn là trade-off giữa:

- throughput
- pause time
- latency
- memory footprint

## Code example

```java
import java.util.ArrayList;
import java.util.List;

public class Main {
    static final List<byte[]> CACHE = new ArrayList<>();

    public static void main(String[] args) {
        byte[] block = new byte[1024];
        // object is strongly reachable from both local variable and static cache
        CACHE.add(block);

        // removing one path is not enough
        block = null;

        // if no other strong reference exists, the byte[] becomes eligible for GC
        CACHE.clear();
    }
}
```

Ở ví dụ này, object không sống theo biến `block`. Nó sống theo toàn bộ graph reference. Chỉ khi mọi strong path biến mất thì GC mới có quyền thu hồi.

## When to use / when NOT to use

Bạn không “gọi GC” như một API business thông thường. Thứ bạn làm là thiết kế object lifetime rõ ràng để GC làm việc đúng.

Dùng mental model này khi:

- debug heap tăng dần
- phân tích memory leak
- đọc GC log
- xem vì sao request xong rồi mà object vẫn còn

Không nên dùng `System.gc()` như cách vá lỗi memory. Nếu app có memory pressure hoặc pause time bất thường, hãy đo bằng profiler, heap dump, và GC log trước khi đụng tới tuning flags.

## How this connects to real Java projects

Trong Spring app, nhiều object sống lâu hơn bạn tưởng vì container giữ reference tới singleton bean, scheduler, cache, hoặc listener.

Điều này làm memory issue trong Spring thường không nằm ở chỗ `new`, mà nằm ở chỗ object bị giữ trong:

- singleton bean field
- in-memory cache không giới hạn
- event listener không cleanup
- `ThreadLocal` trên thread pool

## Gotchas

- GC không thu hồi object còn `reachable`, dù business đã bỏ quên nó.
- `eligible for GC` không đồng nghĩa “bị dọn ngay”.
- `System.gc()` chỉ là hint, không phải guarantee.
- Tuning GC khi chưa đo thường làm vấn đề khó hiểu hơn, không giải quyết gốc rễ.

## Check yourself

- Vì sao câu hỏi đúng không phải là “object được tạo ở đâu” mà là “ai còn giữ reference tới nó”?
- `reachable`, `eligible for GC`, và `collected` khác nhau thế nào?
- Vì sao `block = null` trong ví dụ vẫn chưa đủ để object chết?
- Những `GC root` nào xuất hiện nhiều nhất trong ứng dụng Java thực tế?
- Vì sao cache không giới hạn là vấn đề về lifetime, không chỉ là vấn đề về dung lượng?

## Exercises

### Bài 1: Is Object Eligible For Gc
Độ khó: Dễ

Đề bài:
Cho một boolean `reachable`, trả về `true` nếu object đủ điều kiện để được GC.

Ví dụ 1:
Đầu vào:
```text
reachable = false
```

Đầu ra:
```text
true
```

Giải thích:
Một object không còn reachable thì có thể được collect.

Ràng buộc:
- reachable is a boolean
- Chỉ dùng reachability để quyết định kết quả
- Không mô phỏng GC timing

### Bài 2: Count Reachable Objects
Độ khó: Trung bình

Đề bài:
Cho một array trong đó `true` nghĩa là object còn reachable, hãy đếm số object reachable.

Ví dụ 1:
Đầu vào:
```text
reachable = [true, false, true]
```

Đầu ra:
```text
2
```

Giải thích:
Hai object vẫn còn reachable từ các root.

Ràng buộc:
- reachable là non-null
- 0 <= reachable.length <= 100000
- Chỉ đếm các giá trị `true`

### Bài 3: Classify Object Lifetime
Độ khó: Trung bình

Đề bài:
Cho tuổi của một object tính bằng milliseconds, trả về `"short-lived"` khi tuổi nhỏ hơn `1000`, ngược lại trả về `"long-lived"`.

Ví dụ 1:
Đầu vào:
```text
ageMillis = 250
```

Đầu ra:
```text
"short-lived"
```

Giải thích:
Object này còn trẻ theo threshold được dùng trong bài.

Ràng buộc:
- ageMillis >= 0
- Dùng đúng threshold được nêu trong đề bài
- Chỉ trả về đúng hai label đã liệt kê
