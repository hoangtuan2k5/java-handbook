# TDD

## What is it

`TDD`, viết tắt của `Test-Driven Development`, là cách phát triển code theo vòng lặp nhỏ: viết một test fail trước, viết lượng code nhỏ nhất để test pass, rồi refactor khi đã có safety net.

Điểm cốt lõi của `TDD` không nằm ở việc có test, mà nằm ở thứ tự tư duy. Test được dùng để ép mình làm rõ requirement, giữ scope nhỏ, và nhận feedback liên tục.

### The loop

| Pha | Câu hỏi mình đang hỏi | Kết quả mong muốn |
|---|---|---|
| `red` | hành vi tiếp theo là gì? | test fail đúng lý do |
| `green` | code tối thiểu nào làm test pass? | pass nhanh, không làm quá |
| `refactor` | mình có thể làm code sạch hơn mà vẫn xanh không? | design rõ hơn, ít duplication |

## How I used to misunderstand it

Mình từng nghĩ `TDD` chỉ là viết test gần như cùng lúc với code, miễn sao cuối cùng test có mặt là được.

Hiểu vậy làm mất giá trị lớn nhất của `TDD`. Khi test được viết sau, mình rất dễ hợp thức hóa implementation hiện tại thay vì để test dẫn đường cho design. Vòng `red -> green -> refactor` tồn tại để chống lại thói quen viết quá tay.

Mình cũng từng nghĩ `TDD` là cách tối ưu để gõ code nhanh hơn ngay từ đầu. Thực tế nó tối ưu cho learning loop. Mỗi bước nhỏ hơn, nhưng đổi lại mình ít lạc hướng hơn và thấy sớm khi API đang khó dùng hoặc requirement còn mờ.

## How it actually works

Trong pha `red`, mình viết test nhỏ nhất có thể để mô tả hành vi tiếp theo và nhìn nó fail đúng lý do.

Trong pha `green`, mình chỉ viết đủ code để test đó pass. Không tranh thủ thêm luôn ba behavior khác chỉ vì “đang tiện tay”.

Trong pha `refactor`, mình dọn duplication, cải thiện tên, tách method, hoặc làm design rõ hơn mà vẫn giữ test xanh.

```text
Need next behavior
      |
      v
write one failing test
      |
      v
make it pass with minimal code
      |
      v
clean up design while tests stay green
      |
      v
repeat
```

Điểm quan trọng là fail cũng phải đúng kiểu. Nếu test fail vì setup sai, import sai, hoặc assertion mơ hồ, loop sẽ nhiễu. `TDD` chỉ có giá trị khi mỗi bước fail hoặc pass đều nói với mình điều gì đó có ích về design và requirement.

### TDD decision guide

| Tình huống | `TDD` hợp không? | Vì sao |
|---|---|---|
| validation, parser, utility, business rule | rất hợp | input và output rõ |
| service logic với nhiều branch | hợp | feedback loop giúp giữ scope nhỏ |
| spike khám phá API lạ | chưa hẳn | có thể cần prototype ngắn trước |
| cấu hình framework mù mờ | hợp một phần | nên kết hợp test integration |
| UI hoặc flow còn chưa rõ requirement | yếu hơn | test dẫn đường khó khi behavior chưa rõ |

## Code example

```java
@Test
void returnsMinusOneWhenNoItemMatches() {
    List<String> values = List.of("java", "spring");

    int result = SearchUtils.indexOfFirst(values, "kafka");

    assertEquals(-1, result);
}
```

Nếu đang làm `TDD`, test này sẽ được viết trước implementation. Nó ép mình quyết định sớm contract là khi không tìm thấy thì trả về `-1`, thay vì đợi code xong rồi mới nghĩ tới edge case đó.

Giá trị thật không nằm ở việc có thêm một test. Giá trị nằm ở chỗ test buộc mình chốt contract trước khi thiết kế trượt đi quá xa.

## When to use / when NOT to use

Dùng `TDD` khi:

- requirement đủ nhỏ để đi theo từng bước hành vi
- đang viết business logic, parser, validation, hoặc utility có input và output rõ ràng
- muốn giữ design đơn giản và nhận feedback sớm

Không áp dụng `TDD` một cách giáo điều ở chỗ requirement còn quá mù mờ hoặc đang làm spike kỹ thuật để khám phá API lạ. Khi đó có thể cần prototype ngắn trước rồi mới quay lại vòng test.

Không viết test quá lớn trong một bước. Khi một test đòi hỏi nhiều setup và nhiều assertion khác nhau, mình đang đánh mất nhịp nhỏ vốn là linh hồn của `TDD`.

Không xem `green` là điểm kết thúc. Nếu bỏ qua `refactor`, mình sẽ có code pass test nhưng ngày càng xấu.

## How this connects to real Java projects

Trong Spring app, `TDD` rất hợp với service logic, validation rule, mapper, policy class, hoặc domain helper. Nó giúp mình tách business rule ra khỏi framework boundary, từ đó unit test nhanh hơn và code ít phụ thuộc container hơn.

Với controller, repository, hoặc security config, `TDD` vẫn dùng được nhưng thường phải kết hợp thêm integration test để xác nhận wiring thật. Nói cách khác, `TDD` dẫn đường cho design logic, còn Spring integration test xác nhận môi trường thực thi.

Khi `TDD` gặp khó trong Spring code, đó thường là tín hiệu tốt. Nó đang nói rằng logic của mình dính framework quá chặt hoặc dependency graph đang quá rối.

## Gotchas

- Viết một test bao phủ nhiều behavior cùng lúc làm mất khả năng dẫn đường của bước `red`.
- Thấy test xanh rồi tiếp tục thêm logic lớn mà không quay lại viết test mới là cách rất nhanh để trượt khỏi `TDD`.
- Refactor bị bỏ qua khiến code pass test nhưng vẫn đầy duplication và tên xấu.
- Dùng mock quá nhiều quá sớm có thể đẩy design theo implementation detail thay vì behavior.
- Một test fail vì setup sai chứ không phải vì requirement chưa có sẽ làm vòng feedback bị nhiễu.
- Chạy theo vòng lặp cơ học mà không hỏi “behavior tiếp theo là gì” sẽ biến `TDD` thành nghi thức, không còn là công cụ thiết kế.

## Check yourself

- Vì sao viết test sau khi code xong không còn là `TDD` theo đúng nghĩa?
- Trong pha `green`, vì sao “ít code nhất để pass” lại quan trọng?
- Một test fail như thế nào thì mới hữu ích cho vòng lặp `TDD`?
- Khi nào nên tạm rời `TDD` để làm một spike ngắn?
- Vì sao bỏ qua `refactor` sẽ làm giá trị dài hạn của `TDD` giảm mạnh?

## Exercises

### Bài 1: Choose Next TDD Action

Độ khó: Dễ

Đề bài:
Cho ba cờ `testFails`, `productionCodeCompiles`, và `duplicationObserved`. Trả về `"write-test"` nếu `testFails` là `false` và `productionCodeCompiles` là `false`. Trả về `"make-green"` nếu `testFails` là `true`. Trả về `"refactor"` nếu `testFails` là `false` và `duplicationObserved` là `true`. Trong các trường hợp còn lại, trả về `"done"`.

Ví dụ 1:

Đầu vào:
```text
testFails = true
productionCodeCompiles = true
duplicationObserved = false
```

Đầu ra:
```text
"make-green"
```

Giải thích:
Đang ở pha `red`, nên bước tiếp theo là viết lượng code nhỏ nhất để test xanh.

Ràng buộc:

- Mỗi input là một boolean
- Chỉ trả về `"write-test"`, `"make-green"`, `"refactor"`, hoặc `"done"`
- Không cần chạy test framework thật

### Bài 2: Validate Red Green Refactor Sequence

Độ khó: Trung bình

Đề bài:
Cho `String[] steps`, trong đó mỗi phần tử là `"red"`, `"green"`, hoặc `"refactor"`. Trả về `true` nếu chuỗi bước hợp lệ theo hai rule sau: bước đầu tiên phải là `"red"`, và mọi `"refactor"` chỉ được xuất hiện ngay sau một `"green"`. Nếu không hợp lệ, trả về `false`.

Ví dụ 1:

Đầu vào:
```text
steps = ["red", "green", "refactor", "red", "green"]
```

Đầu ra:
```text
true
```

Giải thích:
Chuỗi này bắt đầu bằng `red` và mọi lần `refactor` đều theo ngay sau `green`.

Ràng buộc:

- `1 <= steps.length <= 100000`
- Mỗi phần tử chỉ thuộc tập giá trị đã cho
- Không cần đánh giá nội dung test thật

### Bài 3: Find First Oversized Change

Độ khó: Trung bình

Đề bài:
Cho `int[] changedLineCounts` và `boolean[] addedProductionCodeBeforeTest`. Mỗi index biểu diễn một bước commit nhỏ trong quá trình làm việc. Hãy trả về index đầu tiên mà `changedLineCounts[i] > 20` hoặc `addedProductionCodeBeforeTest[i]` là `true`. Nếu không có bước nào vi phạm tinh thần thay đổi nhỏ của `TDD`, trả về `-1`.

Ví dụ 1:

Đầu vào:
```text
changedLineCounts = [6, 8, 25, 4]
addedProductionCodeBeforeTest = [false, false, false, false]
```

Đầu ra:
```text
2
```

Giải thích:
Bước ở index `2` thay đổi quá lớn so với vòng lặp nhỏ mà `TDD` nhắm tới.

Ràng buộc:

- `changedLineCounts.length == addedProductionCodeBeforeTest.length`
- `0 <= changedLineCounts.length <= 100000`
- `0 <= changedLineCounts[i] <= 1000000`
